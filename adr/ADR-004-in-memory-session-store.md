# ADR-004: In-Memory Session Store with TTL Eviction

**Status:** Accepted  
**Date:** 2026-04-11  
**Deciders:** Project team

---

## Context

Chat sessions require per-user state: conversation history (list of messages) and a pointer to the active document. A session store is needed. Options include an in-process dict, Redis, SQLite, or a managed cache service.

## Decision

Use a module-level Python dict (`_sessions: Dict[str, dict]`) in `api/session.py`. Sessions are created on first access and evicted passively after 2 hours of inactivity (`SESSION_TTL_SECONDS = 7200`). Eviction runs at the start of each `get_or_create_session()` call.

Documents are registered globally (`_docs`, `_doc_meta`) by `doc_handle` — not per-session. Sessions store only the `active_doc_id` pointer.

## Consequences

**Positive:**
- Zero external dependencies — no Redis, no database
- Sub-microsecond lookups
- Simple eviction via passive TTL check

**Negative:**
- **All session state is lost on server restart** — users start fresh
- **Not horizontally scalable** — two replicas have independent session dicts; sticky sessions required for multi-replica deployment (e.g., ACA session affinity)
- Passive eviction means stale sessions accumulate until the next request arrives — potential memory leak under high load
- Global doc registry is process-wide; no tenant isolation

**Scaling path:**
- Replace `_sessions` dict with Redis (or Azure Cache for Redis) to enable stateless replicas
- Replace `_docs` global with a distributed cache or re-ingest from persistent storage

**Invariants:**
- `session_id` is a UUID generated by the frontend (`uuid()` in `app.js`)
- Session TTL resets on every request touching that session
